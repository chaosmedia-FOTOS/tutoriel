Tutoriel / Utiliser git

Par Massaki Archambault

Catégorie Programmation

Date 16/02.15

Le but de ce tutoriel est de comprendre et utiliser git, un système de versionage très pratique. Git permet à plusieurs développeurs de travailler sur le même code en même temps, de garder un historique des modifications et de centraliser le projet sur un serveur distant, doublant ainsi comme un backup. Il est totalement possible d'utiliser git pour versionner un projet seulement en local, mais il est généralement utilisé avec un service tel que Github ou Bitbucket. Pour la suite de ce tutoriel, lorsque je mentionne le serveur, je me réfère à ce service.

Il existe des dizaines d'interface graphique pour git. Ils ont tous leurs avantage, leurs inconvénients, leur plateforme, popularité, etc. Donc au lieu d'expliquer commet se servir de chaque interface graphique individuellement, ce tutoriel se concentrera sur les commandes console. Il est toujours possible d'accéder à la console à partir de l'interface graphique et les noms de commande ne change pas. Pour faire un push, par exemple, simplement cliquer sur le bouton push

Dans les exemples ci-dessous, je vais parfois faire référence à origin. origin désigne le serveur et il est configuré à l'aide de la commande suivante.

git remote add origin https://lien_vers_le_depot.git

L'équivalent dans l'interface graphique est peut être trouvé dans les options du dépot, mais il est généralement configuré à la création.

Le workflow d'un projet git

Il y a 4 étapes à suivre lorsque l'on se sert de git. La première étape est de faire un pull, ou récupérer les modifications effectué par les autres développeurs sur le serveur. Pour faire un pull, on a besoin de fournir l'url du dépôt git, le nom d'utilisateur du service ainsi que le mot de passe qui lui ai associé.

git pull origin master

La deuxième étape est de d'écrire le code, le tester, etc. comme on fait lorsqu'on développe normalement.

La troisième étape est de faire un commit. Un commit est fait dans une étape du développement où un morceau important est terminé et que le code est stable, c'est à dire raisonnablement sans bug. Il est très important de ne pas faire de commit sur un code instable. Un commit est accompagné d'un message qui décris les modifications effectué au code. La commande add permet de sélectionner quel fichiers sont ajouté dans le commit. Dans une interface graphique, c'est souvent un case à cocher.

git add . git commit -m "Message du commit"

La dernière étape est de faire un push. Un push est lorsqu'on envoie tous ses commits au serveur. Il n'est pas nécessaire de faire un push après chaque commit.

git push origin master
